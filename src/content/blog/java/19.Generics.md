```java
public class demo {
    public static void main(String[] args) {
```

## Wild-Cards in Generics
Generic -> break parent child relationship
Animal -> Dog (example Dog is child)
so can do `Animal a=new Dog();`
### List
`interface List;` it is with many array like operation
`ArrayList` -> concert class implements List
```java
List<Integer> list=new ArrayList<>();
```
if make a list of animal can we put a Dog in the list
```java
List<Dog> dogs=new ArrayList<>();
List<Animal> animals=dogs; // wrong/error
```
breaks parent child relation
```java
Dog is-a Animal
List<Dog> is-a List<Animal> // WRONG 

Generic<A> is Not subtype of Generic<B> 
```
Why it will break parent child relation, because
- it breaks Type-safety
- Reading is not a problem as can read Dog as a Animal
- if added a Animal in a List of Dog => if animal has less method compare to dog thus, will leads to problem.
This property is called invariant
```java
Generics are invariant
	If A is child of B,
	Generic<A> is-not child of Generic<B>
```
problem
```java
Dog[] dogs =new Dog[10];
Animal[] animals=dog;

animals[3]=new Animal(); // run time error
```
example

## Type erasers
Does JVM knows generics
- no, so how it handles it
- T is erased from byte-code
Rule to remove
- if no bound -> replace with `Object`
- if bound -> replace with bound
	- if `extends Number` -> replace T with `Number`
- insert cast automatically -> 
	- as Object to integer
	- can't have `classCastException` (as already tested by compiler)

limitation of generics
- can't do instance of check
```java
// if(l instanceOf List<String>) // ERROR
```
as at runtime all generic info gone so List of Integer,String everything becomes simple `List`
- method can't be overloaded on base of generic
```java
void print(List<String> l){}
// void print(List<Integer> l){} ERROR REDCULARED
```
as generic info lost

- Compiler generated bridge method
```java
class par<T>{
	T get(){
		return null;
	}
}
class child extends par<String>{
	@Override
	String get(){ // not override
		return "Hello";
	}
}
```
compiler makes it
```java
class par{
	Object get(){
		return null;
	}
}
class child extends par<String>{
	String get(){ // not override
		return "Hello";
	}
	// JVM made a bridge method
	Object get(){
		get();
	}
}
```
#### Why type erasers
- because generics came in java 5
- so not made new JVM -> backward compatibility
- generics will be a compiler only test
#### Generic not support primitive why ?
- can't have `int` why only need `Integer`
- `Object` class is not parent of primitive data type -> can't replace with `Object` in compile time thus, can't
- or moreover it is part of OPPs and primitives are to basic for that.
